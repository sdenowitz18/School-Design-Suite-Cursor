Product Spec: Component Snapshot / Overview Tab
1) Purpose and goals

The Snapshot tab provides a fast, high-confidence understanding of a component’s “shape” without requiring deeper navigation.

It should answer four user questions:

What is this? (definition + purpose)

What’s included / how is it organized? (high-level structure + sequence)

How does time work? (daily vs block + basic dosage)

Who does it serve and how many students does it reach? (scope + participation)

Snapshot is a single screen that uses progressive disclosure to show different fields depending on:

Component Type: STEM, Humanities, Wayfinding, Well-being, Cross-cutting

If Component Type is STEM or Humanities: Level = Course vs Subject

Course = collection/set (e.g., Math)

Subject = a specific course (e.g., Algebra I)

Snapshot is not the place to define full designed experience; it should remain “overview” level and link users to deeper screens later.

2) Entry logic and progressive disclosure
Step 1: Component Type (required to render template)

User selects one:

STEM

Humanities

Wayfinding

Well-being

Cross-cutting

Step 2: Course vs Subject (only if STEM or Humanities)

If Component Type is STEM or Humanities, user selects:

Course (collection/set)

Subject (specific course)

After selections are made, the screen renders the appropriate Snapshot template.

3) Screen structure (applies to all templates)

Snapshot is organized into four sections in a single scrollable view:

Definition & Purpose

High-Level Structure

Who it serves / Scope

Participation / Reach (manual entry for now)

Each section can be collapsible if needed for half-screen contexts, but the default should keep Definition & Purpose expanded.

4) Template A: STEM / Humanities — Course Snapshot
A1) Definition & Purpose

Goal: define what the “course collection” is and what it’s for.

Fields

Component Name

Component Set Name (optional; use when this course is part of a named set)

Component Description (scaffolded entry)

Prompt: “In 1–3 sentences, what is this course collection and why does it exist?”

Optional helper prompts (collapsible):

“What should students reliably get from this?”

“How does it differ from current/previous practice?”

Level: Course (display only)

Component Type: STEM or Humanities (display only)

Primary Outcomes (L3) (tag input)

Examples: Algebra, Geometry

A2) High-Level Structure

Goal: show what’s inside the course collection and the broad sequence + time model.

Fields

What’s included (tag list or simple list)

Guidance: “List the subjects/courses included (e.g., Algebra I, Geometry, Algebra II).”

Note: This is not required to map directly to subcomponents; it’s an overview list.

Course Sequence (ordered list)

Display supports an “arrow” preview (Algebra I → Geometry → Algebra II)

Time Model at a Glance

Daily vs Block (two-option selector)

Total time / week

Days per week

Minutes / class

A3) Who it serves / Scope

Goal: clarify reach across grades and major student group targeting.

Fields

Grades Served (range)

Student Groups (multi-select)

Examples: All Students, Honors, Intervention, Pathway

Optional: Placement notes (short text)

A4) Participation / Reach (manual for now)

Goal: capture basic participation and composition until data connections exist.

Fields

# Students participating

% of student body

% advanced vs on-level vs intervention (manual breakdown)

% in each included class/subject (manual breakdown)

Example: % Algebra vs % Geometry vs % Algebra II

Optional: Other composition notes (text)

Optional UI enhancement (ok to implement simply)

A small chart (bars) that reflects the manual breakdown values.

8) Lite acceptance criteria

User can select Component Type (and Course vs Subject when required) and Snapshot updates immediately to the correct template.

Snapshot communicates the four core answers: what it is, how it’s structured, time model, who it serves/reach.

Non-traditional components are understandable without relying on “obvious” labels because Purpose + Where it shows up are present.

Participation/Reach is editable manually and supports a future move to data-connected display without redesigning the section.

Infrastructure Elements can be added in the four categories and remain visible in Snapshot as scannable “requirements.”
